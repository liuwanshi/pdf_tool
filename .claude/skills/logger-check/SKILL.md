# 日志规范性检查技能

对指定范围的代码进行系统化日志审查，确保日志体系完整、可追踪、可倒查。

## 触发条件

- 用户显式调用 `/logger-check` 或 `logger-check`
- 用户说"检查日志"、"日志审查"、"logger review"、"可观测性检查"
- 用户要求做上线前的日志完整性校验

## 日志分类体系

本项目的日志按用途分为三个层级，对应不同的关注点和存储策略：

### 1. console.out — 主流程日志

记录系统正常运行过程中的关键节点，用于了解系统状态、追踪业务流程。

| 场景 | 必须记录 | 示例 |
|------|---------|------|
| 应用启动/停止 | 启动时间、配置摘要、监听端口 | `app: 服务启动 127.0.0.1:5000` |
| 模块初始化 | 模块名、初始化耗时、关键参数 | `PaddleOCR 初始化完成 (耗时 6.2s)` |
| 后台任务生命周期 | 任务ID、类型、开始/完成/失败 | `OCR 任务 abc123: 开始 (3 页)` |
| 定时任务执行 | 任务名、执行耗时、处理数量 | `文件清理完成: 删除 12 个过期文件` |
| 配置变更 | 变更项、新旧值 | `MAX_WORKERS 从 3 调整为 5` |

**要求**：使用 `INFO` 级别，关键节点覆盖率 ≥ 90%。

### 2. error.out — 异常日志

记录所有异常和错误，确保每个故障都有完整的上下文用于排查。

| 要求 | 说明 |
|------|------|
| **捕获即记录** | 每个 `except` 块必须有日志，严禁空 catch |
| **堆栈必须完整** | 使用 `logger.exception()` 或 `traceback.format_exc()` |
| **上下文必须充分** | 记录触发异常的输入参数、当前状态 |
| **用户可感知** | 对外返回的错误信息应与日志关联（建议用 trace_id） |

**反面示例**（严禁）：
```python
try:
    do_something()
except Exception:
    pass  # 空 catch，异常被吞掉，出了事无法排查
```

**正面示例**：
```python
try:
    do_something(file_path)
except Exception as e:
    logger.exception(f"处理文件失败: {file_path}")  # 自动附带堆栈
    raise  # 或者转换为用户友好的错误返回
```

### 3. userlog.out — 用户请求日志

记录每个 HTTP 请求的完整链路，用于追踪用户操作、排查线上问题。

| 位置 | 必须记录 | 级别 |
|------|---------|------|
| **入口 (Before)** | 接口名/PATH、请求方法、关键参数（脱敏后） | INFO |
| **出口 (After)** | 接口名、响应状态码、耗时(ms)、结果摘要 | INFO |
| **异常出口** | 同上 + 异常类型和简要原因 | ERROR |

**推荐方式**：使用 Flask 的 `before_request` / `after_request` 钩子或自定义装饰器统一处理，避免每个路由方法中手写重复代码。

## 检查维度

### 维度一：分类完整性（权重 30%）

检查三种日志是否齐全：

- [ ] console.out: 应用启动、模块初始化、任务生命周期、定时任务 是否有对应日志
- [ ] error.out: 是否所有 `except` 块都有日志、是否有空 catch
- [ ] userlog.out: 是否每个 API 端点都有入口和出口日志

### 维度二：格式规范性（权重 25%）

检查日志格式是否统一、信息是否完整：

| 检查项 | 标准 |
|--------|------|
| 时间戳 | 每条日志必须含时间戳（`logging.basicConfig` 的 `%(asctime)s`） |
| 级别标识 | 含 `[%(levelname)s]` 标记 |
| 模块/函数名 | 含 `%(name)s` 标识日志来源 |
| 关键上下文 | 日志消息中必须包含足够的上下文（task_id、文件名等） |
| 中文可读性 | 日志消息使用中文描述，便于非技术人员理解 |

### 维度三：链路完整性（权重 25%）

检查关键业务流程是否可追踪：

- [ ] **用户请求链路**：入口 → 处理 → 出口，每个环节都有日志串联
- [ ] **后台任务链路**：创建 → 开始 → 进度更新 → 完成/失败，全程可追踪（通过 task_id）
- [ ] **子进程链路**：启动 → 运行中 → 退出，子进程状态可监控
- [ ] **异常链路**：抛出点有日志、捕获点有日志、返回给用户的信息与日志关联

### 维度四:运维友好性（权重 20%）

检查日志系统的可维护性：

- [ ] **日志滚动策略**：是否按时间或大小滚动，避免单文件无限增长
- [ ] **日志级别控制**：是否支持通过配置切换 DEBUG/INFO/WARNING 级别
- [ ] **敏感信息脱敏**：日志中是否避免了密码、Token、身份证号等敏感数据
- [ ] **日志输出目标**：是否支持同时输出到文件和控制台

## 执行流程

### 第一步：确定范围

询问用户要检查的文件或目录。如果未指定，默认检查 `app.py`、`routes.py`、`tasks.py`、`ocr_worker.py`（项目核心流程文件）。

### 第二步：逐文件四维检查

1. **分类完整性轮**：扫描异常处理（grep `except`）、API 路由（grep `@app.route`）、后台任务函数，逐一对照三种日志的记录情况
2. **格式规范性轮**：检查 `logging.basicConfig` 配置、逐条日志的消息格式
3. **链路完整性轮**：从用户请求入口开始，沿调用链追踪，检查每个环节的日志埋点
4. **运维友好性轮**：检查日志配置、文件管理策略

### 第三步：输出报告

按分级输出，标注文件路径和行号。

## 补充建议（报告中应主动评估）

除上述四维检查外，还应评估以下增强项：

1. **结构化日志**：建议使用 JSON 格式输出日志（`python-json-logger`），便于接入 ELK/Loki 等日志平台
2. **Trace ID**：建议每次请求生成唯一 trace_id，贯穿整个请求链路，方便跨模块关联
3. **慢请求告警**：建议在 after_request 中判断耗时超过阈值（如 3s）时输出 WARNING
4. **关键指标埋点**：OCR 页数、压缩前后体积、任务成功率等业务指标应定期汇总输出
5. **日志采样**：高频轮询接口（如 `/api/tasks` 每 2s 一次）应降级为 DEBUG 或做采样，避免日志淹没
