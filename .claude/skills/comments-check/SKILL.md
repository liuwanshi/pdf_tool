# 代码注释检查技能

对指定文件或目录的代码进行三轮逐维度注释质量检查，最终输出分级报告。

## 触发条件

- 用户显式调用 `/comments-check` 或 `comments-check`
- 用户说"检查注释"、"注释审查"、"review comments"、"代码可读性"
- 用户要求合并前做代码质量把关

## 检查维度

### 维度一：覆盖率（权重 40%）

检查注释是否"够多"，标准：

| 级别 | 标准 |
|------|------|
| 模块级 | 每个 `.py` 文件头部必须有模块级 docstring，说明本文件职责 |
| 函数级 | 每个函数/方法必须有 docstring，说明：用途、参数含义、返回值、异常 |
| 类级 | 每个 class 必须有 docstring，说明类的职责和使用场景 |
| 逻辑块级 | 关键流程分支（if/else/循环/try-except）前应有行内注释说明意图 |
| 常量/配置 | 非自明的常量定义应有行尾注释解释其含义和单位 |

**量化参考**：注释行数应大约占代码总行数的 25%~35%（即 10 行中约 3 行注释、7 行代码），不要求精确匹配，用于发现极端缺乏或过度注释的文件。

检查时自底向上扫描：
1. 统计每个文件的总行数、注释行数、注释占比
2. 列出没有 docstring 的函数/类
3. 找出 10 行以上无任何注释的"纯代码块"

### 维度二：准确性（权重 35%）

检查注释是否"说对了"，逐函数验证：

- **功能描述 vs 实际代码**：docstring 描述的功能是否与函数体实现一致
- **参数说明 vs 实际用法**：docstring 中参数名、类型、用途描述是否正确
- **返回值说明 vs 实际返回**：返回值类型和含义描述是否准确
- **废弃/过时注释**：注释提到已不存在的变量、函数、参数、文件路径
- **拷贝残留**：注释中有明显是从别处复制过来的、与当前代码无关的内容

对每个不匹配项，给出"注释说的"和"代码做的"对比。

### 维度三：小白友好度（权重 25%）

站在新手视角评估注释是否"看得懂"：

- **术语解释**：专业术语（如"GIL"、"WAL 模式"、"render_mode=3"）首次出现时是否有解释
- **隐含前提**：注释是否存在依赖读者已有背景知识才能理解的表述
- **行内注释质量**：关键行内注释是否说明了"为什么这样做"，而非仅复述"做了什么"
- **缩写说明**：非通用的缩写变量名是否有注释解释其全称

**反面示例**（小白看不懂）：
```python
# render_mode=3 插入隐形文字
page.insert_text(point, text, render_mode=3)
```
**正面示例**（小白友好）：
```python
# 添加隐形但可被搜索/选中的文字层
# render_mode=3：PyMuPDF 的不可见渲染模式，文字对用户透明但 PDF 阅读器可搜索
page.insert_text(point, text, render_mode=3)
```

## 执行流程

### 第一步：确定范围

询问用户要检查的文件或目录。如果用户未指定，默认检查当前分支相对于 `master` 的所有变更文件（`git diff master --name-only`，仅 `.py` 文件）。

### 第二步：逐文件三轮检查

对每个文件进行三轮独立的检查，不要混合维度：

1. **覆盖率轮**：先统计行数，找缺失 docstring 的函数/类，标记纯代码块
2. **准确性轮**：逐函数比对 docstring 与实际代码实现
3. **友好度轮**：从新手视角通读注释，标记难懂的内容

### 第三步：分级归类

检查结果按严重程度归为三级：

| 级别 | 标记 | 标准 | 示例 |
|------|------|------|------|
| 🔴 必须修复 | critical | docstring 缺失、注释内容错误、严重误导 | 无 docstring 的函数、参数名写错 |
| 🟡 建议修改 | warning | 注释过于简略、术语未解释、覆盖不足 | docstring 只写了一句话、关键逻辑无注释 |
| 🟢 仅供参考 | info | 表述可更清晰、可以提�的优化建议 | 用词不够通俗、缩写建议展开 |

## 报告输出格式

```
# 注释检查报告

## 概览
- 检查文件数：N
- 🔴 必须修复：X 处
- 🟡 建议修改：Y 处  
- 🟢 仅供参考：Z 处
- 总体评价：[一句话总结]

## 🔴 必须修复
### [文件路径:行号] 问题标题
- 维度：覆盖率 / 准确性 / 友好度
- 描述：[具体问题]
- 代码：[问题代码片段]
- 建议：[修复建议]

## 🟡 建议修改
...

## 🟢 仅供参考
...
```

## 注意事项

- docstring 使用中文书写（本项目规范）
- 单行 docstring 用 `"""xxx"""` 格式，多行用标准多行格式
- 不要对明显自解释的代码（如 `x = 1 + 2`）要求注释
- 注释检查不涉及代码逻辑正确性，那是 code review 的职责
- 配置文件（如 `config.py`）的注释要求标准相同
